Structured reduction of the original data

1. Goal of the reduction

The original Capital Bikeshare dataset is very large and very detailed. It contains single rental trips, which means that each row describes one individual bike ride. For our project, this level of detail is not ideal, because our goal is not to predict a single trip, but to predict rental demand. Based on the ideas from the lecture and the exercise notebooks, the first important step is to transform the raw data into a dataset that is easier to understand, easier to analyze, and suitable for machine learning.

The general idea was therefore:
- reduce the size of the dataset,
- keep the information that is most relevant for prediction,
- remove redundant or unnecessary variables,
- create a clean final file that can directly be used for model training.

2. From trip-level data to station-day data

The first major simplification was the level of aggregation. Instead of keeping every single rental as one row, we aggregated the data so that each row now represents one station on one day.

This was done because the prediction task was defined at daily station level:
“How many bikes are rented from a station on a given day?”

This step makes the data much smaller and much more suitable for regression. It also follows the logic presented in the lecture, where the data first has to be transformed into a representation that matches the actual machine learning problem.

3. Restricting the data to the most relevant stations

The full system contains around 900 stations. For a first clean project version, we decided to focus on the 20 most frequently used start stations.

This reduction was made for several reasons:
- the dataset becomes much smaller and easier to handle,
- the most active stations provide the largest amount of data,
- the final dataset is more stable because rare stations with very little data are excluded,
- the first models can be built and interpreted more easily.

This is a practical form of data reduction and problem scoping. Instead of trying to model everything at once, we first focus on the part of the system where the data is strongest.

4. Defining the target variable

At an earlier stage, the dataset still contained several count variables:
- total_rentals
- classic_bike_count
- docked_bike_count
- electric_bike_count
- and even more detailed counts split by user type

After reflecting on the modeling task, we simplified this again and kept only:
- total_rentals

The reason is that the total number of rentals is our main prediction target. The more detailed bike-type counts were removed from the final train-ready file because they are directly related to the total count. In fact, the total rentals are the sum of these counts. If we kept them as input variables while trying to predict total rentals, the model would already receive the answer indirectly. This would create target leakage.

So the target was simplified to one clear output:
- total_rentals

This makes the task easier to explain, easier to evaluate, and methodologically cleaner.

5. Adding external information

After aggregating the rental data, we merged it with daily weather information. Weather variables are important because bike usage is strongly influenced by weather conditions.

We also created time-related variables from the date, because date strings are not directly useful for machine learning models. Instead of keeping the raw date as text, we transformed it into numerical features.

This follows the lecture idea that the final input of a machine learning model should be structured, numeric, and meaningful for the learning task.

6. Simplifying the date information

The original date variable was not kept in the final model dataset. Instead, it was transformed into a smaller set of useful time-related features.

At first, several time variables were created, such as:
- month
- weekday
- year
- day_of_year
- week_of_year
- time_idx

After further simplification, we kept only:
- month
- weekday
- year
- time_idx

These were chosen because they capture the most important time patterns:
- month captures seasonality,
- weekday captures weekly behavior,
- year captures long-term yearly differences,
- time_idx captures the overall trend over time.

The other date-derived variables were removed because they partly carried overlapping information and would make the dataset more complex without a clear benefit in the first model.

7. Correlation analysis and the heatmap

After the first daily dataset was created, we analyzed the predictor variables using a correlation matrix and a heatmap.

This step is strongly connected to what was discussed in the lecture and in the exercise notebooks:
- explore the data before modeling,
- identify redundant predictors,
- reduce multicollinearity,
- keep the dataset interpretable.

The heatmap showed that several weather variables were very strongly correlated with each other. This means that different columns were often carrying nearly the same information.

The strongest groups were:

Temperature block:
- temp
- tempmax
- tempmin
- feelslike
- feelslikemax
- feelslikemin
- dew

Solar and light block:
- solarenergy
- solarradiation
- uvindex
- sunrise-related variables
- sunset-related variables
- daylight-related variables

Wind block:
- windspeed
- windgust

Precipitation block:
- precip
- precipcover
- precipprob

The heatmap made it clear that many of these features were redundant.

8. Why correlated features were reduced

Keeping many highly correlated variables can create problems, especially for simple and interpretable models such as linear regression. If several columns carry almost the same information, the model becomes harder to interpret, and the coefficients may become unstable. This is exactly the kind of issue that was discussed in the lecture under collinearity and redundant predictors.

Because of this, we reduced the weather feature set by choosing one useful representative from each highly correlated group.

Examples:
- tempmax was kept instead of keeping many similar temperature variables,
- windspeed was kept instead of also keeping windgust,
- sunset_minutes was kept instead of keeping several light-related variables,
- uvindex was kept as a compact measure of sunlight conditions.

This means that the final features still describe the weather conditions well, but without unnecessary duplication.

9. Why we did not use PCA

We also considered using PCA, because PCA is presented in the lecture as a method to reduce dimensionality when variables are correlated. However, we decided not to use PCA in our main pipeline.

The first reason is interpretability. Our goal is not only to get a prediction, but also to understand which factors are related to bike demand. With PCA, the original features are transformed into new artificial components. These components are harder to explain, because they are combinations of many variables. In our project, it is easier to justify a model that uses understandable features such as temperature, precipitation, windspeed, weekday, or month.

The second reason is that the lecture notes also mention some disadvantages of PCA. PCA is sensitive to scaling, it can lead to information loss, and the new components are more difficult to relate back to the original variables. For our current stage, we wanted a simpler and more transparent preprocessing pipeline.

The third reason is that we had already reduced the strongest redundancy by using the heatmap and correlation analysis. Since we were able to choose a smaller and clearer set of original variables, PCA was not necessary for the first version of the model. We may still use PCA later as an additional experiment, but not as the main preprocessing method.

10. Final feature choices

The final training dataset keeps only numeric features and one target variable.

The features are:
- start_station_id
- tempmax
- humidity
- precip
- precipcover
- cloudcover
- windspeed
- visibility
- sealevelpressure
- uvindex
- sunset_minutes
- month
- weekday
- year
- time_idx
- snow
- snowdepth

The target is:
- total_rentals

This feature set is simpler than the original dataset, but still contains the most important information about:
- station identity,
- weather conditions,
- seasonal and weekly patterns,
- long-term time development.

11. Why only numeric features were kept

The final train-ready dataset contains only numeric values.

This decision was made because:
- most machine learning models work more directly with numeric inputs,
- numeric data is easier to scale, compare, and analyze,
- it avoids extra preprocessing in the first modeling step,
- it creates a clean and consistent file for training and evaluation.

Text variables such as station names were removed from the training file, but we kept a separate mapping file that links station_id to station name. This way, the final model file stays clean, while interpretation remains possible.

12. Final result

The final reduced dataset is not just a random smaller version of the original data. It is the result of a structured preprocessing pipeline:

1. Start from the raw trip-level dataset.
2. Restrict the problem to the 20 most used start stations.
3. Aggregate single rentals into daily station-level demand.
4. Merge the rental data with weather data.
5. Transform the date into useful numeric time features.
6. Use a correlation matrix and a heatmap to identify redundant predictors.
7. Remove strongly overlapping variables.
8. Keep only numeric input variables and one clear target variable.

This process produced a dataset that is much better suited for machine learning. It is smaller, cleaner, easier to justify, and closer to the actual prediction goal of the project.
